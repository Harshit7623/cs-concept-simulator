## What stack vs heap solves

Programs need memory for two fundamentally different kinds of data: function call state (arguments, return addresses, local variables) and data that outlives the function that created it. The stack handles the first with minimal overhead; the heap handles the second with flexibility.

## How they work

The **stack** is a contiguous region of memory managed by a pointer (the stack pointer `SP`). Calling a function decrements `SP` by the size of the new frame; returning increments it. No bookkeeping is needed — allocation and deallocation are one instruction each.

The **heap** is a region managed by the allocator (`malloc`/`free` or `new`/`delete`). The allocator maintains free lists or lookup tables to find a suitably sized block. Allocation is **O(n)** in worst case; fragmentation can waste space.

## Pseudocode

~~~c
void funcA(void) {
    int x = 10;                        /* on the stack */
    int *ptr = (int *)malloc(32);      /* 32 bytes on the heap */
    funcB(ptr);
    free(ptr);                         /* must free manually */
}

void funcB(int *buf) {
    int y = 20;                        /* on the stack */
    /* buf points to heap memory */
}

int main(void) {
    funcA();
    return 0;
}
~~~

## Common pitfalls

- **Stack overflow**: deep recursion or large stack allocations exceed the fixed stack size.
- **Dangling pointer**: returning a pointer to a stack-allocated variable — the memory is reclaimed when the function returns.
- **Memory leak**: forgetting to `free()` heap memory; the block remains allocated until the process exits.
- **Double free**: calling `free()` on an already freed pointer causes undefined behavior.
- **Use-after-free**: accessing heap memory after it has been freed.
